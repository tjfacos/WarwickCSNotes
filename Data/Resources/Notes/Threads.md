# Threads

> [!note]- Describe threads.
> - A unit of CPU execution.
> - Single threaded process — one chain of execution, runs each line sequentially.
> - Share much in common with parent — code, data, heap, signals, files.
>     - (+) Easier resource sharing between threads.
>     - (+) Less overhead than process → more scalable.
>     - Contains things to be **managed and run code independently** (thread ID, PC, registers and stack).

# Multithreading

> [!note]- Describe multithreading types and Amdahl’s law.
> - **Concurrency** — more than 1 task can make progress.
>     - (+) Makes systems more responsive - less waits.
> - **Parallelism** — processing done simultaneously (can be split via task or data).
>
> **Amdahl’s law:**
>
> - Speedup from multithreading a process.
> - $speedup \leq \frac{1}{S + \frac{1-s}{N}}$
>     - 1 = time taken before parallelising
>     - S = time to run serial part (0 ≤ S ≤ 1)
>     - $\frac{1-S}{N}$= time taken to run parallelisable part with N cores

> [!note]- Describe multithreading models for a webserver.
> - One thread per request:
>     - Large overhead creating each thread.
>     - Large number of threads can burden system.
> - Thread pool:
>     - Faster - less overhead.
>     - Request may have to wait in queue if all workers busy.
>     - Hard to decide number of workers in pool.

# User and Kernel Threads

> [!note]- Describe user and kernel level threads.
>
> Kernel level threads:
>
> - Used to provide services and run user processes.
> - Kernel can schedule threads on different CPUs.
>
> User level threads:
>
> - Exist within user processes.
> - **Must be associated with kernel-level thread to be executed**.

> [!note]- Describe the user-kernel thread mappings.
>
> **1-1 model:**
>
> - Kernel aware of all threads so user process can rely on kernel to manage threads.
> - Expensive as creating user threads involves kernel.
> - User process limited by thread management policy by OS.
>
> **N-1 model:**
>
> - Less overhead in creation of threads.
> - More flexibility in thread management.
> - Multiple user level threads cannot run in parallel.
> - Process blocks when 1 user thread blocks
>
> **N-N model:**
>
> - Can run in parallel and no blocking issues.
> - Programmer can choose how many kernel and user threads to use.

# Signal Handling

> [!note]- Describe signal handling.
> - Sync - signals generated by process (normally sent to only causing thread).
> - Async - externally generated signals, e.g. user created.
> - User defined signal handler can override default handler.
>     - Can deliver to specific or all threads.
>     - Some signals should be delivered to all threads (like SIGINT).
